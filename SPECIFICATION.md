# 毕业论文 Word 转 PDF 批量转换工具 — 需求规格说明书

> 版本：1.0
> 日期：2026-03-26
> 状态：草案

---

## 一、项目概述

### 1.1 项目背景

高校教务管理中，每届毕业生需提交毕业论文的电子版归档。学生提交的文件通常为 Word 格式（`.doc` / `.docx`），而归档或审阅往往要求统一为 PDF 格式。手动逐一转换耗时且易出错。本项目旨在开发一款**跨平台可视化桌面软件**，帮助教务人员一键批量完成毕业论文的 Word → PDF 转换。

### 1.2 核心目标

- 提供简洁的图形化界面，降低使用门槛
- 支持批量转换整个班级/年级的毕业论文
- 智能识别子文件夹中的毕业论文文档，忽略其他无关 Word 文件
- 跨平台运行（Windows / macOS / Linux）

---

## 二、术语定义

| 术语 | 说明 |
|------|------|
| **根文件夹** | 用户选择的顶层文件夹，包含所有学生的子文件夹 |
| **学生文件夹** | 根文件夹下的每个子文件夹，通常以学生姓名命名，也可能是学号、姓名+学号等 |
| **毕业论文文档** | 学生文件夹中需要转换的目标 Word 文件（`.doc` / `.docx` / `.wps`） |
| **无关文档** | 学生文件夹中存在但不需要转换的其他 Word 文件（如开题报告、任务书、文献翻译等） |

| 角色 | 说明 |
|------|------|

| **指导教师** | 偶尔使用。可能需要将所指导学生的论文批量转换后打印或提交 |

## 四、典型文件夹结构分析

### 4.1 标准结构

```
根文件夹/
├── 张三/
│   ├── 张三_毕业论文.docx          ← 需要转换
│   ├── 开题报告.docx               ← 不需要转换
│   └── 任务书.doc                  ← 不需要转换
├── 李四/
│   ├── 基于深度学习的图像识别研究.docx  ← 需要转换（以论文题目命名）
│   └── 文献综述.docx               ← 不需要转换
└── 王五/
    └── 毕业设计论文_王五_终稿.docx   ← 需要转换
```

### 4.2 常见变体与异常情况

| 编号 | 情况 | 说明 | 处理策略 |
|------|------|------|----------|
| V1 | 子文件夹以学号命名 | 如 `20220101/` | 正常处理，不影响识别 |
| V2 | 子文件夹以"学号_姓名"命名 | 如 `20220101_张三/` | 正常处理 |
| V3 | 子文件夹包含嵌套子目录 | 如 `张三/论文/终稿.docx` | 需递归搜索或限定搜索深度 |
| V4 | 论文文件名无统一规范 | 可能叫"毕业论文"、"论文终稿"、"thesis"、论文题目等 | 需要智能识别策略 |
| V5 | 存在多个版本 | 如 `论文_v1.docx`、`论文_v2.docx`、`论文_终稿.docx` | 默认取最新修改时间或提供选择 |
| V6 | 文件夹中只有一个 Word 文件 | — | 直接选取该文件 |
| V7 | 文件夹中没有 Word 文件 | — | 跳过并警告 |
| V8 | 包含 `.doc`（旧格式）或 `.wps`（WPS 格式） | 兼容 Office 97-2003 格式及 WPS 格式 | 均需支持转换 |
| V9 | 文件被加密/损坏 | — | 跳过并记录错误 |
| V10 | 根文件夹下直接存在 Word 文件（无子文件夹） | — | 提示用户确认是否也需要转换 |
| V11 | 空的学生文件夹 | — | 跳过并警告 |
| V12 | 文件名包含特殊字符 | 如空格、括号、中文等 | 正确处理，不报错 |
| V13 | 同名 PDF 已存在 | 目标位置已有同名 PDF | 提供覆盖/跳过/重命名策略 |

## 五、功能需求

### 5.1 核心功能

#### F1：选择根文件夹

- 用户通过文件选择对话框选取根文件夹
- 支持拖拽文件夹到窗口
- 选择后自动扫描并展示文件夹结构

#### F2：智能识别毕业论文文档

软件需从每个学生文件夹中自动识别出毕业论文文档，识别策略按优先级排列：

**规则引擎（默认策略）：**

1. **关键词匹配**（文件名包含以下关键词之一，不区分大小写）：
   - 中文：`毕业论文`、`毕业设计`、`学位论文`、`毕设`、`论文终稿`、`论文定稿`、`最终版`
   - 英文：`thesis`、`dissertation`、`final_paper`
2. **排除关键词**（即使匹配上述关键词也排除）：
   - `开题报告`、`任务书`、`文献综述`、`文献翻译`、`外文翻译`、`中期报告`、`答辩`、`PPT`、`幻灯片`、`成绩`
3. **单文件推断**：若学生文件夹中仅有一个 Word 文件，直接选定
4. **文件大小辅助判断**：毕业论文通常较大（> 100KB），可作为辅助参考
5. **多版本处理**：若匹配到多个候选文件，优先选取文件名含 `终稿`/`定稿`/`最终`/`final` 的版本；若仍有多个，选取最后修改时间最新的

**用户自定义策略：**

6. 允许用户通过设置界面自定义匹配关键词和排除关键词
7. 允许用户手动指定文件名匹配模式（如正则表达式或通配符）

#### F3：预览与手动调整

- 扫描完成后，以**列表/表格**形式展示识别结果：
  - 学生文件夹名
  - 识别到的毕业论文文件名
  - 识别状态（✓ 已识别 / ⚠ 多个候选 / ✗ 未找到）
- 对于**多个候选**的情况，用户可通过下拉框手动选择
- 对于**未找到**的情况，用户可手动浏览并指定文件
- 用户可逐条勾选/取消勾选，决定哪些需要转换
- 提供"全选"/"全不选"快捷操作

#### F4：批量转换

- 点击"开始转换"按钮执行批量 Word → PDF 转换
- 转换过程中显示：
  - 总进度条
  - 当前正在转换的文件名
  - 已完成数 / 总数
  - 预计剩余时间（可选）
- 支持**取消**正在进行的转换任务
- 转换异常时不中断整体流程，跳过失败文件并记录

#### F5：输出管理

- **默认输出位置**：PDF 文件生成在原 Word 文件所在目录
- **可选输出位置**：用户可指定统一的输出目录，PDF 文件按学生文件夹名组织：
  ```
  输出目录/
  ├── 张三/
  │   └── 张三_毕业论文.pdf
  ├── 李四/
  │   └── 基于深度学习的图像识别研究.pdf
  └── ...
  ```
- **可选**：输出到统一目录时，提供 PDF 文件重命名规则配置：
  - `{学生文件夹名}_毕业论文.pdf`
  - `{原文件名}.pdf`（默认）
  - 自定义模板

#### F6：转换结果报告

- 转换完成后展示结果摘要：
  - 成功数量
  - 失败数量及失败原因列表
  - 跳过数量及原因
- 支持将报告导出为 `.txt` 或 `.csv` 文件

### 5.2 辅助功能

#### F7：设置/偏好

- 关键词配置（匹配关键词、排除关键词）
- 默认输出位置设置
- 文件搜索深度设置（默认 2 层）
- 同名 PDF 已存在时的处理策略（覆盖 / 跳过 / 重命名）
- 语言切换（中文 / 英文，预留国际化接口）

#### F8：历史记录（可选，低优先级）

- 记录每次批量转换的时间、文件夹路径、结果概要
- 方便用户回溯操作

## 七、技术选型建议

> 以下为建议方案，最终选型需在技术设计阶段确认。

### 7.1 桌面应用框架

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Electron + React/Vue** | 跨平台成熟、生态丰富、UI 灵活 | 包体较大（~100MB+），内存占用高 | ★★★★ |
| **Tauri + React/Vue** | 包体小（~10MB）、性能好、安全性高 | 依赖系统 WebView，兼容性需验证 | ★★★★★ |
| **Flutter Desktop** | 统一渲染引擎、跨平台一致性好 | 桌面端生态尚不够成熟 | ★★★ |
| **PyQt / PySide** | Python 生态、开发快 | 打包体积大、分发不便 | ★★★ |

**推荐方案：Tauri 2 + Vue 3 / React**
- 包体小、启动快，适合工具类应用
- Rust 后端性能优秀，适合文件处理任务
- 前端使用现代 Web 技术，UI 开发效率高

### 7.2 Word → PDF 转换引擎

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **LibreOffice Headless** | 格式兼容性最好、免费 | 需安装 LibreOffice（约 500MB）、启动慢 | ★★★★ |
| **Pandoc + LaTeX** | 开源、可定制 | 格式保真度有限、需安装 LaTeX | ★★ |
| **python-docx + reportlab** | 纯 Python | 复杂格式支持差 | ★ |
| **unoconv / unoapi** | 基于 LibreOffice、命令行调用简单 | 本质仍依赖 LibreOffice | ★★★ |
| **Microsoft Office COM（仅 Windows）** | 格式完美还原 | 仅 Windows、需安装 Office | ★★★ |

**推荐方案：LibreOffice Headless 模式**
- 格式还原度最高（仅次于 Microsoft Office）
- 跨平台可用
- 可通过内嵌或引导安装解决依赖问题
- 考虑使用连接池/常驻进程优化批量转换性能

### 7.3 补充方案

- Windows 平台可优先检测系统是否已安装 Microsoft Office，若有则使用 COM 接口转换（格式最完美）
- macOS 平台可尝试使用系统内置的 NSDocument API 作为备选
- LibreOffice 作为跨平台兜底方案

## 九、异常处理策略汇总

| 异常场景 | 处理方式 | 用户提示 |
|----------|----------|----------|
| Word 文件损坏/无法打开 | 跳过，记录错误 | "文件损坏，无法转换：[文件名]" |
| Word 文件被密码保护 | 跳过，记录错误 | "文件已加密，请解除密码后重试：[文件名]" |

| 转换引擎（LibreOffice）未安装 | 引导安装 | 提供下载链接和安装说明 |
| 文件正被其他程序占用 | 跳过，记录错误 | "文件被占用，请关闭后重试：[文件名]" |
| 转换超时（单文件超过 60 秒） | 强制跳过 | "转换超时：[文件名]" |

## 十二、已决议事项

> 以下问题已于 2026-03-26 确认。

### D1：转换引擎 — 内嵌，用户零安装

- 应用必须**内嵌转换能力**，用户首次使用无需自行安装 LibreOffice 或任何外部依赖
- 实现策略：
  - 应用内捆绑精简版 LibreOffice Portable / LibreOffice 核心组件
  - 首次启动时自动解压到应用数据目录（如 `~/.pdf_converter/libreoffice/`）
  - 或采用应用安装包中预置方式，随安装包一同分发
- 各平台适配：
  - **Windows**：优先检测系统已安装的 Microsoft Office（COM 接口），若无则使用内嵌 LibreOffice
  - **macOS**：尝试系统 NSDocument API，兜底使用内嵌 LibreOffice
  - **Linux**：使用内嵌 LibreOffice
- 影响：安装包体积将增大（预计 150–300MB），需在安装包大小和用户体验之间权衡

### D2：识别策略 — 宁多勿漏

- 当智能识别无法明确判断时，采用**"宁可多选让用户取消"**策略
- 具体行为：
  - 若有多个候选文件且无法通过规则引擎分出优劣，**全部勾选**并标记为 `⚠ 多个候选`
  - 用户可在预览列表中取消不需要的文件，或通过下拉框选择正确的一个
  - 若学生文件夹中无任何匹配，但存在 Word 文件，将最大的 Word 文件作为候选（标记为 `? 待确认`）
- 设计原则：宁可让用户多看一眼取消，也不要漏掉需要转换的论文

### D3：并发度 — 自适应机器配置

- 批量转换时的并发度需**根据目标机器配置动态调整**
- 实现策略：
  - 自动检测 CPU 核心数和可用内存
  - 默认并发数 = `min(CPU核心数 / 2, 4)`，确保不过度占用系统资源
  - 提供设置项允许用户手动调整并发数（范围 1–8）
  - 内存不足时（可用内存 < 1GB）自动降为串行处理
- 每个并发 worker 使用独立的 LibreOffice 实例/连接，避免竞争

### D4：超大文件特殊处理

- 对超过 **100MB** 的 Word 文件需做特殊处理
- 处理策略：
  - 扫描阶段：在预览列表中**标记警告**（如 `⚠ 大文件 256MB`），提醒用户该文件转换可能较慢
  - 转换阶段：
    - 超大文件单独排队，**不计入并发池**，独占一个转换进程以避免内存溢出
    - 转换超时阈值放宽至 **5 分钟**（普通文件为 60 秒）
    - 转换前检查磁盘剩余空间是否 ≥ 文件大小 × 3（留足临时文件空间）
  - 转换完成后：若生成的 PDF 大小异常（与源文件差异超过 10 倍），标记为可疑并提醒用户核查

### D5：自动更新 — 内置

- 应用需**内置自动更新机制**
- 实现策略：
  - 基于 Tauri 2 内置的 Updater 插件（`@tauri-apps/plugin-updater`）
  - 启动时后台检查更新，发现新版本后以**非侵入式通知**提示用户
  - 用户可选择"立即更新"或"稍后提醒"
  - 更新过程：下载差量包 → 校验签名 → 用户确认 → 重启应用完成更新
  - 设置中提供"检查更新"手动触发按钮
  - 更新服务器：GitHub Releases 或自建 CDN
- 版本号规范：遵循语义化版本（SemVer），如 `1.0.0`、`1.1.0`、`2.0.0`

### D6：支持 WPS 格式

- 应用需支持 **WPS Office 生成的文件格式**
- 支持范围：
  - `.wps` 文件（WPS 文字专有格式）
  - `.wpt` 文件（WPS 模板格式，低优先级）
  - WPS 生成的 `.docx` / `.doc` 文件（通常已兼容，无需额外处理）
- 实现策略：
  - LibreOffice 已原生支持打开 `.wps` 格式，可直接利用
  - 文件扫描阶段需将 `.wps` 纳入识别范围（与 `.doc` / `.docx` 同等对待）
  - 关键词匹配和排除规则同样适用于 `.wps` 文件
- 影响：
  - 文件类型过滤条件更新：`.doc` / `.docx` / `.wps`
  - UI 中文件类型说明需相应调整
